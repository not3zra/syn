# Decision Log

Every decision made during planning, with the hackathon choice and what would change if time/simplicity were not a constraint. Only decisions that would *meaningfully* improve the product if redone are listed.

---

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 1 | Session ID source | Auto-generated time buckets (10-min windows) | Agent-provided explicit session IDs | Time bucketing is a heuristic — it splits legitimate long-running tasks and merges unrelated actions. Explicit session IDs from the agent are semantically accurate. |
| 2 | Sequence matching | Ordered pairs (A→B) | N-action chains with gap tolerance | Pairs catch simple patterns but miss longer attack chains or sequences with benign actions interleaved. Gap-tolerant chains are harder to evade. |
| 3 | Cumulative severity | Simple sum, threshold=70 | Decay-weighted sum | Simple sum lets old actions accumulate forever. Decay weighting (recent actions matter more) is a realistic risk model instead of an arbitrary counter. |
| 4 | AI Bootstrap input | One-time introspection | Continuous auto-registration on new tool appearance | One-time setup means stale rules. Auto-registration keeps rules up to date without manual re-runs. |
| 5 | AI Bootstrap prompt | Hardcoded fintech context | Dynamic context from knowledge base / vector store | Hardcoded context only works for one domain. Dynamic context lets AI Bootstrap generate relevant rules for any codebase without editing prompts. |
| 6 | AI Bootstrap review | Minimal table + "Approve All" | Per-tool staged approval with diff view and rollback | "Approve All" is fast but risky. Per-tool approval with diff view gives the human reviewer confidence to accept or revert individual generated rules. |
| 7 | Slack integration | Incoming webhook (post-only) | Full Slack app with interactive approve/deny buttons | Webhooks post notifications but require switching to the UI to act. Interactive buttons let the approver approve or deny from Slack. |
| 8 | Database | SQLite | PostgreSQL | Irrelevant for a single-user demo. PostgreSQL matters when concurrent access, multi-user, and production scale are required. |

## Round 2 — Post-submission-hardening decisions

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 9 | Session ID source (revised) | Agent-managed lifecycle with gateway-issued IDs (session_intent: start/continue/end), time-bucket fallback | Same | Gateway owning the session ID neutralizes the lying-agent attack. Time-bucket fallback preserves backward compat for unmodified agents. |
| 10 | Sequence matching (revised) | N-action subsequence matching, unlimited gap tolerance, agent-wide sliding window | N-action subsequence with configurable gap tolerance | Pairs missed padded attacks. Unlimited gap catches any subsequence match but may add false positives in long sessions. |
| 11 | Cumulative severity (revised) | Agent-wide sliding time window (30 min default), simple sum | Decay-weighted sum within sliding window | Sliding window prevents session-boundary evasion. Decay-weighting would add realism but wasn't needed for the demo. |
| 12 | AI Bootstrap input | Continuous auto-registration on unknown tool with pending review queue | Same, with auto-rollback if human rejects within a window | One-time setup means stale rules. Auto-registration keeps rules up to date without manual re-runs. Pending queue keeps human in loop. |
| 13 | AI Bootstrap prompt | Dynamic context from domain_config.yaml | Dynamic context from vector store over org policy docs | Config file is the right level for a demo. Vector store matters when an org has 100+ policy documents. |
| 14 | Slack integration | Incoming webhook with resolve URL link. Interactive buttons deferred to production. | Full Slack app with interactive approve/deny buttons | Webhook + link is functional. Interactive buttons require OAuth + public endpoint — a demo-day risk. |
| 15 | Auto-registration notification | Flash-on-load message on Pending Approvals tab | Live polling or websocket push for badge count | Flash message is simpler to build and verify — no timing race to test. Generation is already done by BackgroundTasks before user navigates to the tab anyway. |
| 16 | Auto-registration error handling | Error status with "Retry generation" button; pending_rules table tracks generation_attempts counter | Auto-retry with exponential backoff | Manual retry is simpler to reason about and demo — the reviewer can see the error, understand it, and trigger a retry. Auto-retry would silently mask failures. |
| 17 | Bootstrap audit trail | Approve/reject actions logged to decisions table as bootstrap_approved/bootstrap_rejected entries | Dedicated bootstrap audit table | Logging to the existing timeline is simpler and creates a coherent narrative (blocked → generated → approved → governed) in the same view. |
| 18 | Pending rules diff view | Line-based red/green diff over editable YAML | Full side-by-side diff with syntax-highlighted AST comparison | Line-based diff is simple to build and reads clearly for a demo. Side-by-side AST diff is a production UX investment. |

## What this tells you

Twenty decisions separate the 5-day hackathon submission from a production product. The first set (1-3) were the original session-risk differentiator improvements. Round 2 (9-11) revises those same three decisions based on deeper threat modeling. The AI Bootstrap and Slack improvements (4-7, 12-14) improve setup automation and UX. The auto-registration refinements (15-18) harden the design that was previously scoped loosely. The database (8) matters later but not now.

## Round 3 — Post-security-scan residual risks (July 2026)

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 19 | Authentication / RBAC | None — every endpoint is open | Authn + per-role RBAC on all endpoints | `POST /resolve/{entry_id}` executes a tool on approval and `POST /bootstrap/approve*` writes policy YAML to disk with no guard. Anyone who can reach the port can approve escalations or push policy. Required before production. |
| 20 | Session tracking key | `agent_id` supplied by the caller | Cryptographic, server-issued, attacker-unforgeable session token | Rotating `agent_id` per call bypasses all session/sequence/cumulative tracking (Beat-4 pattern, cumulative threshold). Only authentication mitigates this. |
| 21 | Rate-limit client identity | Raw TCP peer (`request.client.host`); `X-Forwarded-For` honored only when `SYN_TRUSTED_PROXY=true` | Trusted-proxy-aware identity with header spoofing rejected | Behind a reverse proxy the raw peer is the proxy, so all clients collapse to one bucket. Enabling `SYN_TRUSTED_PROXY` trades the spoofing risk for correct per-client limits — only safe when the proxy strips/overwrites `X-Forwarded-For`. |
| 22 | Unknown-size request bodies | Rejected with `411 Length Required` (chunked / no `Content-Length`) | Streamed, capped, and drained cleanly | Starlette's `request.stream()` raises `ClientDisconnect` mid-iteration in this uvicorn/Starlette version, which corrupted connections and crashed the worker. Rejecting unknown-size bodies is the robust fix, but it means a *legitimate* chunked client literally cannot call the API (all real clients send `Content-Length`). |
| 23 | SSRF guard on `api_base` | Blocks non-HTTP(S) and any host resolving to private/loopback/link-local | Same, plus an allow-list of approved MCP hosts | `POST /bootstrap/introspect` can no longer fetch `http://169.254.169.254/` or `http://localhost`. Trade-off: a *localhost/internal* MCP server can no longer be introspected via `api_base` — use `manual_schemas` instead. |
| 24 | Bootstrap prompt injection via `manual_schemas` | Accepted (bounded by schema validation + manual approve) | Sandboxed LLM call / signed tool schemas | Tool name/description/params from `manual_schemas` are interpolated into the bootstrap LLM prompt. Impact is bounded: output is schema-validated (`validate_generated_yaml`) and requires human approval before it becomes policy. |
| 25 | `README.md` | Left empty (`# syn` only) | Real project README | The repo's top-level README still describes nothing — onboarding starts from `docs/`. |
| 26 | Decision-log coverage of hardening | This Round 3 entry (added post-scan) | Existing entries for every security decision | The 12 implemented hardening controls (path sanitization, prompt sanitization, startup validation, LLM timeout+fallback, rate limiting, body size, audit retention, request IDs, async LLM, CORS, SSRF guard, no `--reload`) are recorded in `docs/prd.md` but had no decision-log entry until now. |
| 27 | Role of `session_id` / spoofable `continue` | `session_id` is an audit label only; `/intercept` `continue` accepts any active session UUID without verifying ownership; scoring is keyed by `agent_id` and `score_session` never reads `session_id` | Session UUID bound to owner + verified on `continue`; `session_id` usable for decision lookups | The spoofable `continue` path is **harmless**: no security-relevant code reads `session_id` (`get_session_history` at `audit.py:197` is defined but never called; scoring uses `get_agent_recent_history(agent_id)`, `audit.py:175`). The cross-agent "steal" test showed `cumulative_severity:0`/`pattern_matched:false` for agent B **not** because ownership was validated, but because scoring is keyed by the self-reported `agent_id` and `score_session` ignores `session_id` — B saw an empty history purely from its own `agent_id`; the reused UUID was irrelevant. The deeper point — that `agent_id` is an unauthenticated, self-reported field — is **not a new session-lifecycle bug**; it is the same no-authentication limitation already listed in #19, merely observed through this test. For the hackathon build `agent_id` is attacker-controlled by design (#20). |

### What this tells you
Six items (19-24) are accepted risks that become production blockers; **#27 is not a separate blocker** — it is an instance of the broader no-authentication limitation already in #19, observed via the session feature. Two (25-26) are documentation gaps. The single highest-priority item is **#19 (authentication)** — without it, #20 is also unmitigated and the high-impact endpoints stay unprotected.

**Correction on the cross-agent steal test:** it passed, but it proved the wrong thing. The `cumulative_severity:0` / `pattern_matched:false` for agent B is *not* evidence that the system validated session ownership. Scoring is keyed entirely by the caller-supplied `agent_id` (`get_agent_recent_history(agent_id)`), and `score_session` never sees `session_id`. Agent B inherited nothing only because the query used B's own (empty) `agent_id`; the reused session UUID was irrelevant. The genuine isolation boundary is `agent_id` itself — which is unauthenticated (the #19 limitation), not a session-lifecycle bug. Reusing the victim's `agent_id` (not the session UUID) would inherit their full history, but that is just the absence of auth on the gateway, already known. The `session_id` is a cosmetic audit label and the `continue` path is spoofable — but **harmless**, because nothing security-relevant reads `session_id`.

## Round 4 — Decision-tree `data_sensitivity` floor (July 2026)

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 28 | Escalation floor for `data_sensitivity` | Added `data_sensitivity_floor: 70.0` to `decision_tree` in `policy_config.yaml`; `apply_decision_tree` (`engine/decision_tree.py`) returns `escalated` when `data_sensitivity >= 70`, with trigger `decision_tree:data_sensitivity_floor` | Per-factor floors on every risk axis, plus decay-weighted history | A single high-impact factor was being auto-approved. `query_database "SELECT * FROM users"` scored `data_sensitivity=70` but its **weighted** score was only **16.0** (below `escalate_min` 30.0) → approved; `1; DROP TABLE users--` scored ds=100 / weighted 26.5 → also approved. The engine had hard floors only for `severity` (>90→block) and `confidence` (<40→escalate), so sensitive data access slipped through when the blended score stayed low. Adding a `data_sensitivity` floor routes PII/sensitive access (the `users`/`customers`/`accounts` matches, sensitive deletes) to a human even when the weighted score is low. Floor set at **70** (not 60): the only evidence-backed cases are ds=70 (`users`) and ds=80 (`customers`/`accounts`); 70 catches both with zero risk of an untested ds=60 boundary slipping through unreviewed. `send_payment` $3,000 still auto-approves (ds=0) — that is governed by the separate payment policy threshold (> $5,000), an intentional choice, not this floor. |

### What this tells you
The `data_sensitivity_floor` closes the gap where a *blended* low score masked a *single* high-impact factor. It sits in `apply_decision_tree` **after** the session branch (`apply_session_branches` runs first in `engine/evaluate.py`), so Beat-4 pattern matches and cumulative-severity escalations are untouched. Unit suite (`tests/test_risk_engine.py`) was updated: the two weighted/confidence escalation tests now use non-sensitive inputs, a benign `SELECT` stays approved, and a new `test_escalate_query_database_pii` asserts `SELECT * FROM users` → `decision_tree:data_sensitivity_floor`. The full adversarial script (`test_workflow.sh`) is unaffected in its assertions (they check `decision != null`); the behavioral change is that PII/sensitive queries and sensitive deletes now escalate instead of approving.

## Round 5 — Residual risk: confidence as an unbounded, non-decaying trust signal (July 2026)

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 29 | Confidence baseline as a trust signal | `score_confidence` (`engine/confidence.py`) only ever *increases* with use and never decays within the 90-day retention window; the history lookback is **unbounded** per `agent_id` (ramp: 50 no-history → 30 tool-unused → 50 at 1–4 uses → 70 at 5–9 → 90 at ≥10, saturating) | Decay-weighted, authenticated confidence that resets when the authenticated principal or session changes | With sustained access to a single `agent_id`, an attacker can issue many benign actions (e.g., 10+ small `send_payment`s) to drive `confidence` to its 90 saturation. `confidence` enters the weighted score as `(100−confidence)·0.05`, so a higher confidence *lowers* the score and makes a later malicious action more likely to auto-approve — a "slow-trust" build-up. **This is not a separate vulnerability.** It is a secondary consequence of the *same* root cause as #19/#20/#27: `agent_id` is unauthenticated and attacker-controlled, so an attacker can both *inherit* a trusted `agent_id`'s history and *build* trust on one they control. Fixing authentication (#19) resolves this together with #20/#27 — no separate code change is warranted for the hackathon build. **Performance framing (corrected):** the unbounded per-`agent_id` history scan is a *long-term scaling consideration* at high volumes, not a measured or active concern at realistic hackathon / early-product volumes (hundreds to low thousands of rows per agent). It should not be described as currently slow unless query latency has actually been measured and a number can be shown. |

### What this tells you
#29 is documentation of a production-readiness observation, not a new blocker and not a new code task. It is the same unauthenticated-`agent_id` limitation already captured in #19 (and observed again in #20/#27) — the slow-trust path is just that limitation viewed through the confidence baseline. The single fix that closes #19, #20, #27, and #29 together is authentication; treating #29 as its own item would risk double-counting the same root cause in the roadmap.

## Round 6 — Known issue: auto-bootstrap writes into the committed config (deferred, July 2026)

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 30 | Bootstrap approval persistence | Deferred — leave as-is for this cycle; the committed `engine/policy_config.bootstrap.yaml` is periodically reset to `tools: {}` by hand / by `test_workflow.sh` teardown. No code change now. | Write approved bootstraps to a **separate, gitignored runtime override** (`engine/policy_config.bootstrap.runtime.yaml`) and read/merge from that; keep `policy_config.bootstrap.yaml` as an untouched `tools: {}` tracked baseline | Every runtime that approves a generated tool — `/bootstrap/approve`, `/bootstrap/approve/{tool}`, `/bootstrap/approve-all` (and `/admin/reset` writing `tools: {}`) — writes into the **git-tracked** `policy_config.bootstrap.yaml`. So any local/dev run that approves a tool (including `test_workflow.sh`'s `approve-all`, and the pytest bootstrap suite) rewrites the committed file with LLM-generated tools (`unknown_tool_xyz`, `custom_tool`, `test_tool`, …), leaving `git status` dirty and forcing repeated manual resets. It is a correctness/cleanliness bug, **not** a security hole (bootstrap tools are reviewed before approval), but it makes repo state noisy and risks accidentally committing junk tools into the shipped baseline. The fix is small and contained; it was scoped out of this cycle at the user's direction. |

### What this tells you
#30 is a deferred housekeeping item, not a blocker, and is independent of the auth work (#19) — the committed file is simply the wrong persistence target. When picked up: introduce `policy_config.bootstrap.runtime.yaml` (gitignored); repoint `_get_merged_tools` and the four write sites in `gateway/main.py` (the `_sanitize_bootstrap_path` default at line 360, `bootstrap_approve_tool` at 466, `bootstrap_approve_all` at 504, `admin_reset` at 609) to it; update `test_workflow.sh`'s two resets and `tests/test_gateway.py:406`'s path assertion; add `SYN_BOOTSTRAP_RUNTIME_PATH` for hermetic tests. Until then, the committed `policy_config.bootstrap.yaml` must be kept at `tools: {}` (reset it if a run pollutes it).

## Round 7 — Deploy-safety prereqs for public demo (tripwire, not real auth, July 2026)

| # | Decision | Hackathon choice | Ideal (unconstrained) | Why it matters |
|---|---|---|---|---|
| 31 | Demo auth model for a public deploy | Shipped a deliberately-throwaway `X-Demo-Token` tripwire (#34) gating all mutating/introspect endpoints, a randomized per-visit `agent_id` (#33), `POST /admin/reset` (#35), and frontend wiring that bakes the token and auto-sends the header (#36). Tripwire is a **no-op when `DEMO_TOKEN` is unset** (local dev) and the gateway **refuses to start on Fly** (`FLY_APP_NAME` set) if `DEMO_TOKEN` is unset (`gateway/main.py:166`). Gated: `/intercept`, `/resolve/{entry_id}`, `/bootstrap/approve`, `/bootstrap/approve/{tool}`, `/bootstrap/reject/{tool}`, `/bootstrap/approve-all`, `/bootstrap/retry/{id}`, `/bootstrap/introspect`, `/admin/reset`. Open: `/timeline`, `/health`, `/tools`, `/bootstrap/pending`. | Real authentication / RBAC (#19) | A public Fly demo needs a ship-blocking minimum so the open endpoints (#19) cannot be trivially abused by anyone who finds the URL. A full auth system was out of scope, so we shipped a throwaway tripwire: the token is baked into the client bundle (extractable by design) and only proves "you read the demo," not "you are authorized." It raises the bar — randomized `agent_id` stops accidental cross-visit history bleed and the refuse-to-start guarantee forces the tripwire on for the public deploy — without pretending to be security. The real fix remains #19. |

### What this tells you
#31 is the decision behind GitHub issues #33–#36. It is a deployment prereq, not a security boundary: the token lives in the public JS bundle, so treat it as a demo courtesy, not access control. The stale residual-risk notes in `docs/prd.md` ("No authentication/RBAC", "Fresh `agent_id` evasion") are now partially mitigated by #31/#33 — update the PRD to match. Real authentication (#19) and the bootstrap runtime-override split (#30) are still deferred.
